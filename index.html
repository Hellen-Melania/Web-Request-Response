2.2 List of headers

  • HTTP Status Codes
    • 200, використовуються для успішних запитів
    • 300 є для перенаправлень
    • 400, використовуються, якщо виникла проблема із запитом
    • 500, використовуються, якщо виникла проблема з сервером
  • Accept призначений для інформування сервера про типах даних, які підтримуються клієнтом (браузером). У цьому заголовку браузер перераховує, які типи документів він "розуміє".
  • Content-type призначений для ідентифікації типу переданих даних. При цьому заголовок Content-type використовує змінну оточення CONTENT_TYPE. Зазвичай для цього заголовка вказується значення application / x-www-form-urlencoded. Таким чином, вказується формат, в якому всі керуючі символи (тобто символи, які не є алфавітно-цифровими) спеціально кодуються.
  • Content-length містить рядок, в якій записана довжина переданих даних в байтах при використанні методу передачі POST. За заголовком закріплена однойменна змінна CONTENT_LENGTH.Якщо задіюється метод GET, то цей заголовок відсутній, і значить, змінна оточення не встановлюється.
  • Cookie зберігаються всі Cookies. Даний заголовок використовує змінну оточення HTTP_COOKIE. Для встановлення Cookies використовується заголовок Set-Cookie.
  • GET використовує такі змінні:
    • REQUEST_URI - запитуваний ідентифікатор ресурсу;
    • QUERY_STRING - передані сценарієм параметри;
    • REQUEST_METHOD - метод передачі інформації.
    В даному випадку ця змінна буде містити значення GET.
  • Location --> Отримавши заголовок Location разом із зазначеним у ньому URL, сервер негайно переходить за вказаною URL, не чекаючи, поки тіло документа завантажиться:
  • POST використовує ті ж змінні оточення, що і заголовок GET (змінна REQUEST_METHOD містить значення POST). Дані методом POST можна передавати в кінці заголовків.
  • Pragma -- > Даний заголовок використовується для різних цілей, одна з яких - це заборона кешування документа. Приклад заголовка: Pragma: no-cache
  • Server містить назву і версію програмного забезпечення сервера.
  • Referer --> За допомогою цього заголовка можна дізнатися сервер який посилається на нас. Це буває корисно, коли нам необхідно дізнатися зворотні посилання, наприклад, при аналізі ефективності мережевої реклами.
  • User-Agent містить версію браузера, ОС. Наприклад: User-Agent: Mozilla / 5.0 (compatible; Konqueror / 3.0.0-10; Linux).
  • Accept-Language відображає настройки мови за замовчуванням користувача.
  • Accept-Encoding--> Більшість сучасних браузерів підтримують GZIP, і відправиляють це в заголовку. Далі веб-сервер може послати HTML в стислому форматі, що зменшує розмір до 80%.
  • Authorization --> data base64 encoded
  • Content-Encoding зазвичай встановлюється, коли повертається вміст архівом(стислий).
  • Host - ім'я хоста
  • Cache-Control -використовується для вказівки директиви, які повинні дотримуватися всіма механізмами кешування уздовж ланцюжка запиту / відповіді. Наприклад: Cache-Control: max-age=3600, public
  • Content-Disposition інструктує браузер, щоб відкрити файл вікно завантаження, замість того, щоб намагатися розібрати зміст. приклад: Content-Disposition: attachment; filename="download.zip"
  • Etag - використовується для кешування
  • Last-Modified вказує на дату останньої зміни документа, в форматі GMT: Last-Modified: Sat, 28 Nov 2009 03:50:37 GM

3. What is differenct between HTTP version 1.0 and 1.1?

HTTP 1.0 (1994)

При роботі по протоколу HTTP 1.0 з подібним типом з'єднання немає офіційної специфікації. По суті, це доповнення до існуючого протоколу. Якщо браузер підтримує постійні з'єднання, він посилає додатковий заголовок в запиті: Connection: Keep-Alive Потім, коли сервер отримує такий запит і генерує відповідь, він також додає в заголовок відповіді Connection: Keep-Alive Після цього з'єднання не розривається, а залишається відкритим. Коли клієнт відправляє ще один запит, він використовує це ж з'єднання. Так буде тривати до тих пір, поки клієнт або сервер не вирішать, що обмін закінчено, і одна зі сторін завершить з'єднання.

HTTP 1.1 (1996- 2015)

  • Формалізує багато розширень до версії 1.0
  • Підтримує постійнe і конвеєрне з'єднання
  • Підтримка Chunked трансфери, стиснення / декомпресія
  • Підтримка віртуального хостингу
  • Підтримка декількох мов
  • Kорисний для відновлення перерваних передач даних
    HTTP 1.1 є розширення HTTP 1.0. Нижче перераховані чотири основних покращення:
  • Ефективне використання IP-адрес, що дозволяють обслуговувати кілька доменів з однієї IP-адреси.
  • Швидке реагування, тобто веб-браузер може відправляти кілька запитів по одному зв'язку.
  • Додовання підтримки кешу
  • Підтримки фрагментованого кодування
  • підтримка активності з'єднання і таким чином збільшення продуктивності

Відмінності

Найбільша відмінність в тому що HTTP / 1.0 передбачає передачу лише одного запиту і одної відповіді через одне TCP з'єднання, а HTTP / 1.1 - багатьох, тобто для того щоб викачати скажімо п'ять файлів з сервера зовсім необов'язково п'ять разів встановлювати TCP з'єднання з ним

  • Extensibility
  • Caching
  • Bandwidth optimization
  • Network connection management
  • Message transmission
  • Internet address conservation
  • Error notification
  • Security, integrity, and authentication
  • Content negotiation

GET and POST

GET: отримати доступ до існуючого ресурсу. В URL перерахована вся необхідна інформація, щоб сервер зміг знайти і повернути в якості відповіді шуканий ресурс.

Приклад простого запиту: GET http://phpbook.info/

POST: використовується для створення нового ресурсу. POST запит, що зазвичай містить в собі всю потрібну інформацію для створення нового ресурсу.

Приклад простого запиту: POST http://phpbook.info/cgi-bin/test HTTP/1.0

Відмінності

GET передає дані серверу використовуючи URL, коли POST передає дані, використовуючи тіло HTTP запиту. Довжина URL'а обмежена 1024 символів, це і буде верхнім обмеженням для даних, які можна відіслати GET'ом. POST може відправляти набагато більші обсяги даних. Ліміт встановлюється веб-сервером і звичайно дорівнює близько 2MB. Передача даних методом POST більш безпечна, ніж методом GET, так як секретні дані (наприклад пароль) не відображаються безпосередньо в web-клієнта користувача (на відміну від URL, який видно майже завжди).

POST - в основному використовується для того, щоб передавати великі обсяги даних. Наприклад, при завантаженні файлів через Web або при обробці форм. Крім того, метод POST часто використовують для естетичних цілей: справа в тому, що при застосуванні GET - URL сценарій стає досить довгим і незграбним. Передані сценарієм параметри не відображаються у вікні браузера, POST-запит залишає URL без зміни.

5.Use actions on "Network" tab

В цьому розділі показуються всі запити, виконані при завантаженні даної сторінки і її компонентів.

  • Стовпець «Method» містить тип запиту (GET, POST).
  • «Status» - відповідь сервера, він повинен бути 200 ОК. Тут важливо звернути увагу на помилки 404 і 403, які слід усунути. Їх причинами можуть бути неправильні адреси (атрибути src). Редіректи 301 і 302 також бажано усунути, щоб зменшити загальну кількість запитів, а значить і швидкість завантаження сайту.
  • «Type» - тип вмісту, наприклад для веб-документів має бути text / html.
  • «Size / Content» - розмір кожного запитаного компонента.
  • «Time / Latency» - час завантаження.
Загалом за допомогою інструменту Networks можна оцінити і проаналізувати роботу сервера, розміри файлів і постаратися зменшити їх вагу(розмір) де це можливо для пониження навантаження на сервер, а ще виявити і усунути можливі помилки.

Filter, search and ordering by columns - пошук інформації в браузері(required source...)

Network складається з п'яти панелей:

  • Controls - контролює вигляд і функціонал
  • Filters - фільтрування даних в таблиці
  • Overview - показує тимчасову шкалу коли був отриманий ресурс
  • Requests Table - таблиця перераховує всі ресурси, які були вилучені. За замовчуванням сортується в хронологічному порядку, з самих ранніх ресурсів у верхній частині. При натисканні на ім'я ресурсу дає більше інформації про нього.
  • Summary - повідомляє про загальну кількість запитів, кількість переданих даних і час завантаження
